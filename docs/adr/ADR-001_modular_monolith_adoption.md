# ADR-001: 모듈러 모놀리스 채택 (마이크로서비스 기각)

**Status:** Accepted / **Date:** 2026-07-09

### Context
포트폴리오 프로젝트로서 MSA가 "있어 보이는" 유혹이 있으나, 트래픽(피크 10 RPS 미만)과 운영 인력(1인)이 현실 제약이다.

### Options

| 차원 | A. 모듈러 모놀리스 | B. 마이크로서비스 | C. 단순 모놀리스(계층형) |
|---|---|---|---|
| 복잡도 | 낮음 | 높음 (네트워크, 배포, 관측 N배) | 최저 |
| 운영 부담 (1인) | 배포 1개, 로그 1곳 | 서비스 수만큼 증가 | 배포 1개 |
| 장애 모드 | 프로세스 단위 | 부분 장애 + 분산 추적 필요 | 프로세스 단위 |
| 확장 경로 | 절단선 따라 분리 가능 | 이미 분리됨 | 분리 시 대수술 |
| 면접 방어력 | "규모에 맞는 판단" 서사 | "왜 이 규모에 MSA?"라는 역질문 리스크 | 판단 근거가 안 보임 |

### Decision
**A. 모듈러 모놀리스.** Spring Boot(Kotlin) 단일 애플리케이션, 내부를 auth/game/collection/play/stats/import 모듈로 분리. 모듈 간 접근은 공개 인터페이스로만 (Gradle 멀티모듈 또는 Spring Modulith로 강제).

### 기각 이유
- B: 10 RPS에 분산 시스템의 세금(서비스 디스커버리, 분산 트랜잭션, 관측 인프라)을 낼 이유가 없음. 1인 운영에서 새벽 장애 시 디버깅 표면적만 넓어짐.
- C: 지금은 같지만, 도메인 경계가 코드에 없으면 확장 시점에 절단선을 찾는 비용이 큼. 모듈 경계는 지금 공짜로 그을 수 있는 미래 보험.

### 뒤집는 트리거
import 잡이 API 응답 지연에 영향(리소스 경합)을 주기 시작하면 → import 모듈을 별도 워커 프로세스로 분리 (첫 물리 분리).
