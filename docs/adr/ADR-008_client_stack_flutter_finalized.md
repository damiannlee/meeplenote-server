# ADR-008: 클라이언트 스택 — Flutter 확정 (Compose Multiplatform·React Native 기각)

**Status:** Accepted / **Date:** 2026-07-10

### Context
ADR-006에서 "네이티브/크로스플랫폼 앱"까지는 정했으나 구체 스택은 열어뒀다. 이 결정을 지배하는 제약 조건이 이후 대화에서 명확해졌다:
1. **개발자는 프론트엔드를 별도로 학습하지 않고, 클로드 코드(AI 코딩 에이전트)가 클라이언트 코드를 전담 작성한다.** 본인이 직접 프론트 코드를 짜고 디버깅하지 않는다.
2. M1(10초 기록)이 출시 시점부터 카메라 첨부·로컬 DB 임시저장을 요구한다 — 네이티브 기능 의존이 초반부터 있다.

이 두 조건 때문에, 애초 유력했던 "백엔드(Kotlin/Spring Boot)와 언어 통일" 논리는 크게 약화된다. 언어 통일의 이점은 **개발자 본인이 코드를 읽고 짤 때** 강력한데, AI가 전담 작성하면 그 이점은 부차적(코드 리뷰 시 읽기 편함) 수준으로 축소되기 때문이다.

### Options

| 차원 | A. Flutter | B. React Native (Expo) | C. Compose Multiplatform |
|---|---|---|---|
| AI 코드 생성 신뢰도 | 높음 — 학습 데이터·실전 패턴 성숙 | 높음 — 가장 오래되고 흔한 스택 | 낮음 — iOS Stable화가 2025.5로 아직 1년 남짓, 학습 데이터 얇음 |
| 툴체인 단순성 (AI 자가 수정 루프 적합성) | 최고 — 단일 툴체인, JS/네이티브 계층 분리 없음 | 중간 — Expo 매니지드는 단순하나 네이티브 모듈(카메라·로컬DB) 연동 시 계층이 벌어짐 | 낮음 — Gradle 멀티플랫폼 빌드 체인 복잡성을 JetBrains 공식 문서도 인정 |
| M1 요구(카메라·로컬DB) 충족 | 성숙한 1st-party급 플러그인(camera, sqflite/drift) | expo-camera·expo-sqlite로 가능하나 네이티브 계층 노출 시 마찰 증가 | Jetpack 라이브러리(Room 2.7+) 멀티플랫폼 지원 성숙하나 iOS 쪽 실전 사례 상대적으로 얇음 |
| 유명세·채용 시장·자료량 | 매우 큼(2위권) | 가장 큼(생태계 최대) | 니치, 급성장 중이나 자료 얇음 |
| 이 프로젝트에서 유명세의 실익 | 부분적 (직접 채용 지원 목적 아니면 자료량만 의미) | 대부분 무효화 — RN 유명세는 "리액트 아는 웹 개발자"에게 강한 이점인데, 본인은 리액트 배경 없고 직접 개발도 안 함 | 낮음 |
| 백엔드와 언어 통일 이점 | 없음(Dart) | 없음(JS/TS) | 있음(Kotlin) — 그러나 AI 전담 개발 조건에서 실효 크게 축소 |

### Decision
**A. Flutter.** 단일 툴체인으로 클로드 코드의 "수정→빌드→확인" 반복 루프에서 마찰이 가장 적고, M1이 요구하는 카메라·로컬 DB가 성숙한 플러그인으로 안정적으로 커버된다.

### 기각 이유
- **B (React Native)**: 자료량·유명세는 최대지만, 그 유명세의 핵심 원천(리액트 생태계 친숙도)을 이 프로젝트는 활용할 수 없다(직접 개발 안 함, 리액트 배경 없음). 남는 실익은 자료량뿐인데, M1이 요구하는 네이티브 기능 연동 시 JS-네이티브 계층 분리가 AI 자가진단 난이도를 Flutter보다 높인다.
- **C (Compose Multiplatform)**: 애초 가장 유력했던 후보였으나, "AI 전담 개발"이라는 조건이 확정되며 유일한 결정적 장점(언어 통일)이 무력화됨. 반면 단점(빌드 체인 복잡성, 상대적으로 얇은 iOS 실전 사례)은 AI 전담 개발에서 오히려 크게 불리하게 작용. 트래픽·운영 조건과 무관하게 순수히 "개발 방식(AI 전담)"이 이 결정을 뒤집은 사례로 기록.

### Consequences
- (+) 클로드 코드의 반복 개발 루프에 최적화된 선택. 빌드 실패 시 원인 분기가 단순.
- (+) 카메라·로컬 DB 등 M1 필수 기능의 플러그인 성숙도 높음.
- (−) 백엔드(Kotlin)와 클라이언트(Dart)가 언어적으로 분리 — "언어 통일" 포트폴리오 서사 대신 "AI 협업으로 이종 스택 풀스택 완성" 서사로 대체.
- (−) 개발자 본인이 Dart/Flutter 코드를 세밀히 리뷰하기 어려움 — 코드 품질 검증은 클로드 코드의 설명·테스트 커버리지에 의존하는 비중이 커짐. 핵심 로직(결제·인증 등 민감 영역은 없음)에 한해 리스크 낮음.

### 뒤집는 트리거
클로드 코드의 Flutter 코드 생성 품질이 실측에서 기대에 못 미치면(빈번한 빌드 실패, 반복적 동일 오류) React Native로 재검토. Compose Multiplatform은 개발 방식이 "AI 전담"에서 "본인 직접 개발 병행"으로 바뀌는 경우에만 재검토 대상.
