# ADR-009: 모듈 경계 강제 — 패키지 규약 + ArchUnit (Gradle 멀티모듈·Spring Modulith 기각)

**Status:** Accepted (개발자 승인 완료 2026-07-11) / **Date:** 2026-07-11

## Context

ADR-001은 모듈 경계 강제 수단을 "Gradle 멀티모듈 또는 Spring Modulith"로 열어뒀다. 5단계(프로젝트 스캐폴드)에서 이 하위 결정을 확정해야 한다. 결정을 지배하는 제약은 ADR-008과 동일하다: **Claude Code가 코드를 전담 작성하는 개발 방식**에서, 빌드 실패의 원인 분기가 단순할수록 개발 루프가 빠르다.

## Options

| 차원 | A. 패키지 규약 + ArchUnit | B. Gradle 멀티모듈 | C. Spring Modulith |
|---|---|---|---|
| 강제 수준 | 테스트 실패 (빌드 게이트) | 컴파일 실패 (가장 강함) | 테스트 실패 (verify()) |
| 빌드 복잡성 | 없음 — 단일 모듈 유지 | build.gradle.kts × 모듈 수, 의존 그래프 관리 | 없음 (단일 모듈) |
| AI 개발 루프 적합성 | 높음 — 규칙이 테스트 코드에 명시적으로 보임 | 낮음 — 모듈 간 의존 추가마다 빌드 스크립트 수정 왕복 | 중간 — 규약이 프레임워크 관례에 암묵적 |
| 추가 의존성 | ArchUnit (테스트 스코프만) | 없음 | Spring Modulith 런타임 의존 (Boot 4 대응 2.x는 아직 신생) |
| 부가 기능 | 없음 (경계 검사만) | 없음 | 모듈 문서 자동 생성, 이벤트 발행 레지스트리 |

## Decision

**A. 패키지 규약 + ArchUnit.** 규약: `com.meeplenote.<module>.api`(공개 계약) / `com.meeplenote.<module>.internal`(비공개 구현). `ModuleBoundaryTest`가 ① 타 모듈의 internal 접근 금지, ② 모듈 간 순환 의존 금지를 CI에서 강제한다.

## 기각 이유

- **B (Gradle 멀티모듈)**: 컴파일 타임 강제는 가장 강하지만, MVP 규모(모듈 7개, 코드 수천 줄)에서 빌드 스크립트 관리 비용이 이득을 초과한다. 특히 AI 전담 개발에서 "빌드 실패가 코드 문제인지 빌드 구성 문제인지"의 분기가 늘어나는 것이 ADR-008의 선택 기준(툴체인 단순성)과 정면 충돌한다.
- **C (Spring Modulith)**: 목적 적합성은 가장 높은 도구지만, ① 지금 필요한 것은 "경계 검사" 하나뿐이고 그건 ArchUnit 규칙 6줄로 충분하다. ② 부가 기능(이벤트 레지스트리)은 현재 모듈 간 통신이 동기 인터페이스 호출뿐이라 사용처가 없다. ③ Boot 4 대응 버전이 신생이라 AI 학습 데이터가 얇다 — ADR-008에서 Compose Multiplatform을 기각한 것과 같은 이유.

## 뒤집는 트리거

- 모듈 간 통신이 이벤트 기반으로 전환되는 시점(예: "기록 저장 → 통계 갱신"을 비동기 이벤트로 분리) → Spring Modulith의 이벤트 발행 레지스트리가 실익을 가지므로 재검토.
- import 모듈을 별도 워커 프로세스로 물리 분리하는 시점(ADR-001 트리거) → 그 모듈만 Gradle 서브모듈로 추출.
