# ADR-010: 엔티티 식별자 — BIGINT IDENTITY + FK 유지 (UUID PK·모듈 간 FK 제거 기각)

**Status:** Accepted (개발자 승인 완료 2026-07-12) / **Date:** 2026-07-12

## Context

`V1__init.sql`의 모든 테이블 PK는 `BIGINT GENERATED ALWAYS AS IDENTITY`이고, 모듈 경계를 넘는 참조(`plays.game_id → games.id`, `play_players.player_id → players.id` 등)는 DB FK 제약으로 걸려 있다. 이 결정이 명시적으로 기록된 적이 없어 재확인했다.

쟁점은 두 가지였다: ① PK를 UUID로 바꿀 이유가 있는가, ② 모듈러 모놀리스(ADR-001)가 향후 서비스 분리로 갈 가능성을 고려해 모듈 간 FK 제약을 지금 제거해둘 이유가 있는가.

## Options

- **A. 현행 유지** — BIGINT IDENTITY PK, 모듈 간 FK 제약 그대로.
- **B. UUID PK 전면 전환** — 모든 테이블 PK를 UUID로.
- **C. 내부 BIGINT PK + 외부 참조용 `public_id UUID` 컬럼 병행** — 모듈 간 참조는 점진적으로 `public_id`로 이관할 수 있게 보험을 깔아둠.
- **D. 모듈 간 FK 제약만 우선 제거** (PK 타입은 유지).

## Decision

**A. 현행 유지.**

## 기각 이유

- **B (UUID PK 전면 전환)**: FK가 걸린 상태에서 PK 타입을 바꾸는 비용과, FK만 걷어내는 비용은 완전히 다르다. FK 제약 제거는 `DROP CONSTRAINT` 한 줄로 즉시·저비용인 반면, PK 타입 전환(BIGINT→UUID)은 해당 컬럼과 이를 참조하는 모든 FK 컬럼·인덱스를 다시 채워야 하고 API로 노출된 id를 쓰는 클라이언트(Flutter 로컬 DB, 임포트 매칭 로직)까지 손봐야 한다. 데이터가 거의 없는 지금이 가장 싸게 먹히는 시점인 건 맞지만, 그 이득을 정당화할 트리거(아래)가 아직 발생하지 않았다.
- **C (내부/외부 id 병행)**: 서비스 분리 트리거가 오기 전까지는 죽은 컬럼·죽은 매핑 코드로 남아 순수 오버헤드다. 테이블마다 "어떤 상황에 어떤 id를 쓰나"를 판단해야 하는 규칙이 늘고, 이 규칙 위반은 ArchUnit으로도 못 잡는다(DTO가 실수로 내부 id를 노출해도 컴파일·테스트가 통과함). ADR-002·§3 NFR의 "관측된 문제 없이 선제 최적화 금지" 원칙과 정면 충돌하며, 실제 서비스 분리 시 필요한 작업(FK 제거, 이벤트/API 계약 재설계) 전체에서 이 컬럼이 줄여주는 비중도 크지 않아 보험료 대비 보장 범위가 애매하다.
- **D (FK만 우선 제거)**: 모듈 경계 침해를 코드 레벨(ArchUnit)이 아니라 스키마 레벨에서 막으려는 시도인데, ADR-009가 이미 "모듈 경계 = 패키지 규약 + ArchUnit"으로 그 문제를 다루고 있다. FK는 그와 별개로 DB가 공짜로 보장해주는 참조 무결성 수단이며, 이를 먼저 걷어내면 §3 NFR의 "데이터 유실 제로 지향"을 코드에서 직접 재구현해야 하는데 아직 그 대가를 치를 이유(서비스 분리 실행)가 없다. 현재 M1 구현에서 `CollectionOwnershipChecker`가 `collections` 테이블을 임시 직접 조회하는 것도, 모듈 경계를 "코드가 지키고 스키마는 당분간 공유"한다는 실질적 합의가 이미 있다는 방증이다.

## 뒤집는 트리거

- ADR-001의 트리거(import가 API 레이턴시에 영향 → 별도 워커/프로세스로 분리) 또는 ADR-009의 트리거(모듈 간 통신이 이벤트 기반으로 전환)가 실제로 발동해 물리적 서비스 분리가 논의되는 시점 → 그때 분리 대상 모듈부터 FK 제거 + PK 전략(UUID 또는 옵션 C의 `public_id`) 재검토.
- 위 트리거 전에는 재논의하지 않는다.
