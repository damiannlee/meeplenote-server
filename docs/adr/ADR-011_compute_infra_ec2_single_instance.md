# ADR-011: 컴퓨트 인프라 — AWS EC2 단일 인스턴스 (RDS 기각)

**Status:** Accepted / **Date:** 2026-07-21

## Context

서비스를 어디에 배포할지가 지금까지 전혀 결정된 바 없었음(ADR-001~010은 전부 애플리케이션 설계). 목표 트래픽(MAU 1,000, 피크 RPS<10, `04_system_design.md` §1 NFR)에 맞는 컴퓨트 스펙과 앱/DB 배치 구조를 정해야 함. 무료 티어(벤더 불문)를 더 이상 쓸 수 없는 제약이 있어 실비용 기준으로 사이징해야 함.

## Options

- **A. AWS EC2 단일 인스턴스(t4g.small, 2vCPU/2GB)** — 앱+Postgres를 Docker Compose로 한 인스턴스에 동거
- **B. AWS EC2(앱) + RDS(관리형 DB) 분리**
- **C. 저가 VPS(DigitalOcean/Vultr 등) 동일 스펙**
- **D. Naver Cloud Platform 동일 스펙**

## Decision

**A.**

## 근거

- **사이징**: Little's Law(L=λW)로 RPS<10, 요청당 처리시간 100ms 가정 시 평균 동시 요청 1개 수준 → CPU는 피크에도 여유. 메모리는 JVM(~700-1000MB) + Postgres(~250-400MB) + OS/Docker(~200-300MB) 합산 ~1.1-1.65GB로, 2GB 인스턴스에서 300-500MB 안전마진 확보.
- **상시 피크 과금 기댓값**(RPS 10/20을 24시간 유지한다는 보수적 가정): 인스턴스 정액 요금(~$12.3/월)이 청구액 대부분이고, egress만 트래픽에 비례(1배 ~$4.7/월, 2배 ~$9.3/월). 사진이 presigned URL로 스토리지 직행이라 EC2 egress에 안 잡히는 게 이 비용을 낮게 유지하는 핵심 전제(→ ADR-012). t4g.small 베이스라인(vCPU당 20%)이 추정 CPU 사용량(1배 ~7.5%, 2배 ~15%)보다 위라 크레딧 소진/서차지 리스크 낮음.
- **B(RDS 분리) 기각**: 무료 티어가 없는 지금 RDS는 별도 과금 라인(월 ~$15)이 추가되는데, 이 데이터 규모(연 100MB 성장, `04_system_design.md` §1)에서 관리형 DB의 이점(자동 백업·장애조치)을 정당화할 트래픽/가용성 요구가 아직 없음. 앱 컨테이너 옆에 Postgres 컨테이너를 같이 띄우고 `pg_dump` + 오브젝트 스토리지 업로드로 백업하는 게 이 단계엔 충분(ADR-002 정신을 인프라에도 적용, 루트 CLAUDE.md §3 "백업 대상은 PostgreSQL 하나").
- **C/D(VPS/NCP) 기각**: 무료 티어 소멸 이후 AWS EC2 온디맨드 가격이 동급 VPS와 사실상 동일(월 $12대) — 가격 차이가 없으면 취업시장 인지도(IAM/VPC 등 AWS 경험)가 타이브레이커. NCP는 국내 결제·시장 맥락은 있으나 글로벌 인지도가 상대적으로 낮아 이번 프로젝트 우선순위(포트폴리오 최우선, 전역 CLAUDE.md §0)에서 밀림.

## 트레이드오프로 받아들인 제약

- 앱과 DB가 한 인스턴스에 있어 **독립적으로 확장 불가** — 하나가 리소스 포화되면 인스턴스 전체를 올리거나 분리 마이그레이션(다운타임 또는 복제)이 필요.
- 단일 인스턴스라 무중단 배포/이중화 없음 — 리사이즈·패치 시 다운타임 발생. MAU 1,000 초기 단계에선 감내 가능한 리스크로 판단.
- 스케일 아웃(수평 확장) 진입 시 ALB(~$16.4/월) + DB 분리(RDS ~$15/월) 고정비가 한 번에 붙어 첫 스텝 비용이 기존 대비 ~3.5배로 뜀 — 그래서 트래픽이 늘어도 **수직 확장(인스턴스 크기 업)을 먼저 쓰고, 수평 확장은 최대 인스턴스로도 부족하거나 실질적 이중화 요구가 생겼을 때만** 검토.

## 뒤집는 트리거

- 앱 또는 DB 어느 한쪽만 리소스가 포화(CPU 크레딧 상시 소진, 메모리 압박)되어 나머지 하나에 영향을 주기 시작 → 그때 DB를 RDS 또는 별도 인스턴스로 분리.
- 가장 큰 t4g 인스턴스로도 처리량이 부족하거나, 무중단 배포/이중화가 실제 요구사항이 됨 → 그때 ALB + 다중 앱 인스턴스로 수평 확장.
- 위 트리거 전에는 재논의하지 않는다.
