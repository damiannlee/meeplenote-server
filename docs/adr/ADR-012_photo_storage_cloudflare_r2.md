# ADR-012: 사진 오브젝트 스토리지 — Cloudflare R2 (S3 기각)

**Status:** Accepted / **Date:** 2026-07-21

## Context

설계(`04_system_design.md` §5)상 사진은 presigned URL로 클라이언트가 오브젝트 스토리지에 직접 업로드하고, DB엔 `photo_key`만 저장한다(`V1__init.sql`의 `plays.photo_key`). 어떤 오브젝트 스토리지 벤더를 쓸지는 미결이었다. 무료 티어를 쓸 수 없는 전제에서 실비용으로 비교해야 함.

## Options

- **A. Cloudflare R2**
- **B. AWS S3**

## Decision

**A.**

## 근거

- 저장 비용 자체는 두 벤더가 GB당 몇 센트 차이(R2 ~$0.015, S3 ~$0.023)라 이 데이터 규모(월 750MB 증가 가정: MAU 1,000 × 월 5회 기록 × 50% 사진 첨부 × 300KB)에선 무의미한 수준.
- 결정적 차이는 **egress(다운로드) 과금**: S3는 GB당 과금(~$0.09/GB), R2는 무료. S2(기록 카드 공유) 기능이 설계상 "앱 밖 사람들이 카드를 보게 만드는" 성장 채널을 의도하고 있어(`03_mvp_feature_spec.md` S2 서버 스코프 판정) 조회 트래픽이 유저 수보다 빠르게 늘어날 수 있는 구조 — egress-heavy 성장 벡터와 정확히 맞물린다. 현재 MAU 1,000 가정에선 절대 금액 차이가 작지만(장당 평생 조회 10회 가정 시 연 ~$8 수준), 유저당 조회수가 늘어나는 시나리오에서 격차가 선형으로 벌어짐.
- R2가 S3 호환 API라 SDK/코드 수준의 채택 비용이 없음 — 벤더 선택이 아키텍처에 새 제약을 추가하지 않음.
- 부가 이점: R2는 나가는 트래픽도 무료라 **향후 벤더를 다시 바꿀 때(탈출 비용)도 S3보다 유리** — S3였다면 다른 곳으로 이전할 때도 egress 과금이 발생.
- DB엔 `photo_key`만 저장하고 URL을 통짜로 저장하지 않는 기존 설계 덕에, 벤더 교체 시에도 스키마 변경 없이 presign/read 로직만 교체하면 됨 — 이번 결정이 나중에 바뀌어도 전환 비용이 낮다.

## 뒤집는 트리거

- R2 가용성/성능 이슈가 실측으로 확인되거나, AWS 생태계(컴퓨트까지 동일 벤더로 묶어 IAM 하나로 관리, ADR-011)로 통합했을 때의 운영 부담 절감이 egress 비용 절감보다 커지는 시점.
- 위 트리거 전에는 재논의하지 않는다.
